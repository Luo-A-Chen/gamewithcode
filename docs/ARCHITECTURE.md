# 游戏架构设计文档

## 总体架构

采用 **组件式游戏循环** 架构，核心思想：

```
Game Loop
  ├── Input → 处理玩家输入
  ├── Update → 更新所有实体状态（物理、AI）
  ├── Collision → 碰撞检测与响应
  ├── Camera → 更新视口
  └── Render → 绘制画面
```

## 核心模块

### 1. Game（游戏主控）
- **职责**：游戏循环、模块协调、状态管理
- **设计**：持有所有子系统的引用，按固定顺序调用 update/render
- **关键决策**：使用 `requestAnimationFrame` + delta time，保证帧率无关的物理计算

### 2. InputManager（输入管理）
- **职责**：聚合键盘状态
- **设计**：事件驱动（keydown/keyup）→ 维护 Set → 每帧查询
- **优势**：解耦输入事件与游戏逻辑，支持同时按键

### 3. Camera（相机系统）
- **职责**：视口跟随、边界限制
- **设计**：线性插值平滑跟随，支持关卡边界约束
- **可扩展**：添加震动效果、多相机、分屏

### 4. CollisionSystem（碰撞系统）
- **职责**：AABB 碰撞检测、碰撞响应
- **设计**：
  - 实体 vs 地形：基于瓦片遍历，最小穿透轴分离
  - 实体 vs 实体：宽相检测 + 回调
- **性能考虑**：只检测实体附近的瓦片，不做全地图遍历

### 5. Renderer（渲染器）
- **职责**：绘制关卡、UI
- **设计**：分离游戏逻辑与渲染，便于后续替换渲染后端

## 实体系统

```
Entity (基类)
  ├── Player    — 玩家控制、跳跃、重力
  ├── Enemy     — 巡逻 AI、可被踩死
  └── Collectible — 收集物、浮动动画
```

- 所有实体共享 `update(dt)` / `render(ctx)` 接口
- `onCollide(other)` 实现碰撞回调
- `dead` 标记用于延迟清理

### 架构选型：继承 vs ECS

| 维度 | 继承体系（当前） | ECS（备选） |
|------|----------------|------------|
| 复杂度 | 低，适合小项目 | 高，适合大项目 |
| 可组合性 | 中等（继承层级深时变差）| 高（组件自由组合）|
| 性能 | 一般（对象分散）| 优（数据连续、缓存友好）|
| 学习曲线 | 低 | 中高 |

**当前选择继承**的原因：项目体量小（7个世界、~10种实体），继承足够用。
**何时考虑 ECS**：如果实体类型超过 30 种，或需要大量实体（>500）同屏，考虑迁移。

## 关卡系统

- **瓦片地图**：二维数组，数字编码瓦片类型
- **单向平台**：类型 2，只从上方阻挡
- **关卡数据**：当前硬编码，后续可改为 JSON/编辑器导出

## 物理系统

简单的逐帧积分：

```
vy += gravity * dt    // 重力
x += vx * dt          // 水平移动
y += vy * dt          // 垂直移动
```

碰撞后分离并归零对应速度分量。

## 后续扩展方向

| 阶段 | 内容 |
|------|------|
| 1. 基础 | ✅ 移动、跳跃、碰撞、敌人、收集物 |

## 设计原则

1. **模块化**：每个文件职责单一，便于独立修改
2. **ES Modules**：原生模块系统，无需打包工具
3. **渐进式**：先跑起来，再优化和扩展
4. **零依赖**：纯 Canvas API，不引入第三方库
